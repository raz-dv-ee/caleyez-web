# CalEyeZ Web App

Browser-based, on-device food recognition and calorie tracking. The full desktop CalEyeZ pipeline
(two YOLO11l-cls experts + an XGBoost arbiter + BLE weight) runs **inside the phone browser** via
onnxruntime-web. No install, no app store, and no cloud does the inference.

**Live:** https://raz-dv-ee.github.io/caleyez-web/
**Engineering console:** https://raz-dv-ee.github.io/caleyez-web/engineering.html
**Latency bench:** https://raz-dv-ee.github.io/caleyez-web/bench.html

## What it does

1. Camera captures a frame; the central ROI is cropped and preprocessed in code (crop, resample,
   RGBA to RGB, divide-by-255, HWC to CHW). This is classical image processing, done before the CNN.
2. Two ONNX experts run on-device: **Global** (320 px, 132 classes) and **Israeli** (224 px, 13 classes
   + background).
3. A **20-feature XGBoost arbiter**, evaluated in JavaScript (onnxruntime-web has no TreeEnsemble op),
   outputs P(israeli) and routes to one expert. Confidence gates decide if the answer is trustworthy.
4. If the on-device system is unsure, it falls back to **Gemini vision** (cloud) to identify the food.
   If there is no internet, it says so honestly instead of guessing.
5. Nutrition is resolved from a local DB, else from **USDA** through a smart filter, scaled by weight.
   A cooking-method and sub-food picker adjusts calories using multipliers derived from USDA data.
6. Weight comes from a reverse-engineered **SWAN BLE scale** over Web Bluetooth (Android Chrome), or
   manual grams.

Models are **not** stored here. They are hosted in the main `CalEyeZ` repo under `/webmodels/` and
fetched cross-origin (CORS) so this repo stays tiny and deploys in seconds.

## On-device runtime

- **Default: WASM (CPU).** Upcasts the fp16 weights to fp32 at run time, so accuracy matches the
  desktop exactly. Recognition runs fully offline after the first load.
- **Opt-in: WebGPU (`?gpu=1`).** Faster fp16 compute on the GPU, but fp16 rounding can flip a
  borderline prediction on some mobile GPUs, so it is not the default.
- Inference is timed and shown live (`N ms on-device edge`) as proof there is no cloud call.
- ORT tensors are disposed after every run (WASM/GPU memory is not reclaimed by the JS GC).

## Nutrition backend (Cloudflare Worker)

One Worker, routed by HTTP method:

- **`GET /?q=<food>`** queries USDA FoodData Central and returns per-100g `{kcal, pro, carb, fat}`.
  A smart filter rejects junk descriptions (juice, dried, sauce, ...), scores candidates by
  primary-name and token overlap, treats cooking words as stop-words, and penalises ALL-CAPS branded
  entries. `?debug=1` returns the ranked candidate list (used by the engineering console).
- **`POST` `{image}`** sends the ROI to Gemini (`gemini-flash-latest`) for the "unsure" fallback and
  returns the identified food.

### Deploy the Worker

```bash
npm i -g wrangler
wrangler deploy
wrangler secret put USDA_KEY      # free key: https://fdc.nal.usda.gov/api-key-signup.html
wrangler secret put GEMINI_KEY
```

Then set `const USDA_PROXY = "https://<your-worker>.workers.dev"` in `index.html` and
`engineering.html`. A local `.dev.vars` (git-ignored) holds the keys for `wrangler dev`.

## Cooking multipliers (evidence-based)

`cooking_multipliers.json` is generated by `scripts/eval/cooking_multipliers.py` in the main repo. It
queries USDA raw-vs-cooked food pairs and takes the per-100g cooked/raw kcal ratio as the multiplier,
aggregated with a sample count and 95% CI. Because the scale weighs the cooked food, per-100g-cooked
is the correct basis. Result: raw x1.0, boiled x1.04, grilled x1.17, fried x1.47, deep-fried x1.99.

## BLE scale protocol (SWAN)

8-byte notify packet on characteristic `ffb2`:

```
byte:  0      1    2     3           4     5     6      7
role:  header len  unit  status/hi   low   high  magic  checksum
       AC     05   00    (256 carry) low   high  CE     Sum(bytes 2..6) & 0xFF
weight = low + (high | carry) * 256      # do NOT mask bit 0 of the high byte
```

## Compatibility

- **Web Bluetooth (scale)** works on Android Chrome/Edge only. iOS (Safari and Chrome-iOS, both
  WebKit) and Firefox block it, so on iPhone weight falls back to manual grams.
- Needs a modern browser with WebAssembly, ~50 MB model download (cached), and >= 2 GB RAM.
- Camera requires HTTPS (secure context).

## Deploy the app

Static site on GitHub Pages. Push to `main`; Pages serves it. Use `?v=N` to bust the CDN cache when
